# ТЗ для Codex: Android-приложение «Приватная сессия VPN»

## 1. Цель продукта

Нужно создать Android-приложение, которое решает не задачу «сделать VPN полностью невидимым», а задачу **практической приватной сессии**:

1. Доверенные приложения получают интернет через VPN.
2. Все остальные приложения остаются **без сети вообще** во время приватной сессии, если пользователь включил системный режим `Блокировать подключения без VPN`.
3. Приложение работает **из коробки**, без root, без Shizuku, без ADB-команд, без ручных shell-танцев.
4. Приложение должно быть полноценным VPN-клиентом для Android с упором на **Xray / VLESS / REALITY**.
5. Приложение должно иметь встроенный **сканер-проверяльщик**, который помогает понять, насколько текущая конфигурация палится обычным приложением и есть ли практические утечки.

Главная идея продукта:

> Мы не обещаем полную невидимость VPN для всей системы Android. Мы делаем так, чтобы **только доверенные приложения имели сеть через VPN**, а все остальные приложения не могли выйти в интернет и, следовательно, не могли удалённо узнать текущий egress IP и отправить сетевую телеметрию.

---

## 2. Базовые принципы

### 2.1 Что приложение делает

1. Поднимает VPN через `VpnService`.
2. Работает как обычный VPN-клиент и как режим «Приватная сессия».
3. Поддерживает импорт профилей в привычных для Xray-пользователя форматах.
4. Помогает пользователю включить системные Android-настройки, которые усиливают режим приватной сессии.
5. Встроенно проверяет локальные и сетевые утечки.

### 2.2 Что приложение НЕ обещает

1. Не обещает полную невидимость VPN для любого приложения в системе.
2. Не обещает защиту от системных, привилегированных или root-приложений.
3. Не обещает сокрытие факта наличия VPN на уровне всех Android API.
4. Не обещает невозможность локального детекта вообще, если пользователь сам включает рискованные режимы.

### 2.3 Что должно работать «из коробки»

Под «из коробки» понимается:

1. Установка APK.
2. Импорт конфига.
3. Выбор приложений, которым можно дать сеть.
4. Нажатие одной кнопки запуска.
5. Переход пользователя только в штатные системные настройки Android, если нужно включить:
   1. `Всегда включённый VPN`
   2. `Блокировать подключения без VPN`

Никаких обязательных действий через root, Shizuku, ADB, shell и внешние сервисы управления устройством.

## 3. Ключевая модель работы

## 3.1 Два режима

### Режим 1. Обычный VPN

Это стандартный режим клиента.

Что делает:

1. Подключается к выбранному серверу.
2. Может вести весь трафик устройства через VPN.
3. Может поддерживать обычное раздельное туннелирование.
4. Подходит для сценария «мне просто нужен VPN».

### Режим 2. Приватная сессия

Это основной режим проекта.

Что делает:

1. Поднимает VPN только для приложений из списка доверенных приложений.
2. Приложения вне списка доверенных приложений остаются без сети, если пользователь включил `Блокировать подключения без VPN`.
3. Явно показывает, какие приложения имеют интернет, а какие нет.
4. Запускает встроенные проверки утечек и побочных каналов.
5. Может безопасно поднимать локальный SOCKS-inbound как power-user режим, но только в защищённой форме.

---

## 4. Техническая суть приватной сессии

## 4.1 Как это реализуется

В режиме «Приватная сессия» приложение использует:

1. `VpnService.Builder.addAllowedApplication()` для списка доверенных приложений.
2. Явную настройку DNS внутри VPN.
3. Запрет обхода VPN, то есть отсутствие `allowBypass()`.
4. Системную Android-настройку `Блокировать подключения без VPN`.

Результат должен быть таким:

1. Доверенные приложения имеют сеть через VPN.
2. Недоверенные приложения не имеют сети вообще.
3. Недоверенные приложения не могут просто сходить на внешний endpoint и увидеть egress IP.

## 4.2 В чём отличие от обычного split tunneling

Обычное раздельное туннелирование решает вопрос маршрута: что идёт через VPN, а что мимо него.

Режим «Приватная сессия» решает вопрос доступа к сети:

1. доверенные приложения получают сеть через VPN,
2. остальные приложения не должны иметь сети вообще.

Это ключевая ценность проекта.

## 5. Ограничения Android, которые надо честно учитывать

## 5.1 Что реально можно гарантировать

1. Только доверенные приложения используют VPN.
2. При включённом `Блокировать подключения без VPN` приложения вне списка доверенных приложений остаются без сети.
3. Если не использовать `allowBypass()`, штатный обход VPN приложениями не должен работать.
4. Если локальный SOCKS не поднят или поднят безопасно, другие приложения не смогут просто использовать его как боковую дверь.

## 5.2 Что гарантировать нельзя

1. Нельзя обещать полное сокрытие факта наличия VPN для всей Android-системы.
2. Нельзя обещать невозможность локального детекта через все возможные API.
3. Нельзя обещать защиту от системных приложений и приложений с повышенными правами.

## 5.3 Правильная ценность продукта

Правильная формулировка для UI и документации:

> Во время приватной сессии только явно разрешённые приложения имеют интернет через VPN. Остальные приложения остаются без сети и не могут удалённо проверить текущий egress IP или отправить сетевую телеметрию.

## 6. Поддерживаемые протоколы и форматы

## 6.1 Протоколы MVP

Обязательно поддержать:

1. **VLESS + REALITY**
2. **VLESS + TLS / XTLS / Vision**, если это возможно для выбранного ядра
3. **Trojan**
4. **Shadowsocks**
5. **VMess**

## 6.2 Протоколы второй очереди

Желательно подготовить архитектуру под:

1. **Hysteria2**
2. **TUIC**
3. **WireGuard**
4. **Цепочки через SOCKS-outbound**

## 6.3 Форматы импорта

Приложение должно уметь импортировать:

1. `vless://...`
2. `vmess://...`
3. `trojan://...`
4. Подписку по URL
5. Файл профиля JSON / YAML / TXT
6. QR-код
7. Буфер обмена
8. Ручное создание профиля
9. **Raw Xray JSON** формата `inbounds/outbounds/log/routing/dns`

## 6.4 Важное требование по raw Xray JSON

Нужно поддержать конфиги формата, похожего на такой:

- `inbounds[]`
- `outbounds[]`
- `log`
- `routing`
- `dns`

То есть пользователь должен иметь возможность вставить или импортировать Xray-совместимый JSON-конфиг, где, например:

1. inbound содержит локальный `socks`
2. outbound это `vless`
3. `streamSettings.security = reality`

При импорте нужно делать нормализацию:

1. Разобрать JSON.
2. Определить поддерживаемые inbound/outbound.
3. Не падать на неизвестных секциях.
4. Не логировать секреты.
5. Показать пользователю, какие части профиля импортированы полностью, а какие частично.

### 6.4.1 Политика по DNS при импорте Xray JSON

Важно: **отсутствие блока `dns` в импортируемом Xray JSON считается допустимым**.

Это не должно ломать импорт и не должно считаться ошибкой профиля.

Правильная логика такая:

1. Если в raw Xray JSON есть верхнеуровневый блок `dns`, приложение импортирует его и показывает пользователю, какие DNS пришли из профиля.
2. Если блока `dns` нет, приложение всё равно успешно импортирует профиль.
3. Если блока `dns` нет, приложение автоматически подставляет безопасный DNS по умолчанию на уровне собственного Android VPN-клиента.
4. Значения DNS по умолчанию должны быть изменяемыми в настройках приложения.
5. В UI нужно явно показывать источник DNS:
   1. `Из профиля`
   2. `По умолчанию приложения`
   3. `Пользовательский`
6. Пользователь должен иметь возможность переопределить DNS, даже если он пришёл из импортированного профиля.
7. Отсутствие `dns` в старых Amnezia/Xray-профилях должно считаться нормальным сценарием совместимости.

## 6.5 Политика по локальному SOCKS

Важно: **мы не запрещаем SOCKS вообще**.

Правильная политика такая:

1. По умолчанию локальный SOCKS выключен.
2. Пользователь может включить локальный SOCKS как power-user режим.
3. В безопасном режиме локальный SOCKS должен быть:
   1. только на `127.0.0.1`
   2. с логином и паролем
   3. по умолчанию на случайном порту
   4. по умолчанию со случайными учётными данными на каждую сессию
4. В UI нужно явно показывать статус:
   1. `Выключен`
   2. `Безопасный`
   3. `Небезопасный`
5. В режиме «Приватная сессия» небезопасный локальный SOCKS должен давать критическое предупреждение.

## 7. Архитектура приложения

## 7.1 Технологический стек

1. Язык: **Kotlin**
2. UI: **Jetpack Compose**
3. Архитектура: **MVVM + UseCases + Repository**
4. Хранилище: **Room + DataStore**
5. DI: **Hilt**
6. Логи: локальный ring buffer + экспорт в файл
7. Сборка: стандартный **Android Studio Gradle project**

## 7.2 Ядро VPN

Codex должен выбрать движок, который на Android стабильно поддерживает **VLESS + REALITY**.

Требования к VPN-движку:

1. Совместимость с Xray-профилями.
2. Возможность строить исходящий туннель для `VpnService`.
3. Поддержка строк `vless://...` и raw JSON.
4. Возможность запускать локальный SOCKS в безопасном режиме.
5. Минимум магии и максимум предсказуемости в Android lifecycle.

## 7.3 Модули приложения

### 7.3.1 Менеджер профилей

Отвечает за:

1. хранение профилей
2. импорт ссылок и файлов
3. парсинг подписок
4. выбор активного узла
5. обновление подписок

### 7.3.2 Контроллер VPN-движка

Отвечает за:

1. старт/стоп ядра
2. подготовку конфигурации
3. привязку движка к `VpnService`
4. настройку DNS
5. управление локальным SOCKS

### 7.3.3 Контроллер приватной сессии

Это главный модуль.

Отвечает за:

1. список доверенных приложений
2. сбор allowlist для `VpnService`
3. проверку системных требований
4. запуск и остановку приватной сессии
5. самопроверки безопасности

### 7.3.4 Защита от утечек

Специальный модуль `Leak Guard`.

Отвечает за:

1. проверку отсутствия опасных локальных inbound по умолчанию
2. проверку, что `allowBypass()` не используется
3. проверку, что DNS явно задан внутри VPN
4. проверку режима локального SOCKS
5. самодиагностику при старте сессии

### 7.3.5 Лаборатория сканера VPN

Отдельный модуль для проверки гипотез и утечек.

Отвечает за:

1. симуляцию обычного приложения, которое пытается понять, есть ли VPN
2. проверку внешнего IP
3. проверку split tunneling
4. проверку локального SOCKS
5. сбор человекочитаемого отчёта

## 8. Логика запуска приватной сессии

Когда пользователь нажимает кнопку **«Запустить приватную сессию»**, приложение должно выполнить следующий сценарий.

### Шаг 1. Предпроверка

1. Выбран активный профиль сервера.
2. В списке доверенных приложений есть хотя бы одно приложение.
3. У приложения есть разрешение на поднятие VPN.
4. Проверено, включён ли `Всегда включённый VPN`.
5. Проверено, включён ли `Блокировать подключения без VPN`.
6. Если какой-то пункт не выполнен, показать понятный экран с инструкцией и кнопкой перехода в системные настройки.

### Шаг 2. Подготовка конфигурации

1. Собрать список доверенных приложений.
2. Подготовить allowlist для `addAllowedApplication()`.
3. Подготовить VPN-конфиг и DNS.
4. Подготовить локальный SOCKS, если он включён пользователем.
5. Прогнать `Leak Guard` перед стартом.

### Шаг 3. Поднятие VPN

1. Запустить ядро.
2. Поднять `VpnService`.
3. Привязать движок к tun.
4. Проверить, что сессия действительно активна.

### Шаг 4. Пост-проверка

1. Проверить внешний IP для доверенного приложения.
2. Проверить, что недоверенные приложения без сети.
3. Проверить DNS внутри туннеля.
4. Проверить локальный SOCKS.
5. Показать итоговый статус.

### Шаг 5. Остановка сессии

1. Остановить `VpnService`.
2. Остановить ядро.
3. Очистить временные параметры локального SOCKS, если он был включён.
4. Сохранить лог сессии.

## 9.11 Экран лаборатории сканера VPN

Это отдельный важный экран.

Задача:

1. Эмулировать поведение обычного приложения, которое пытается понять, есть ли VPN.
2. Дать пользователю и разработчику ответ: насколько текущая конфигурация практично палится.

Сценарии:

### Проверка Android API

Нужно уметь:

1. Получить активную сеть через `ConnectivityManager`.
2. Получить `NetworkCapabilities`.
3. Получить `LinkProperties`.
4. Зафиксировать, видны ли признаки VPN.
5. Показать, какие DNS, маршруты и интерфейсные свойства доступны обычному приложению.

### Проверка внешнего IP

Нужно уметь:

1. Узнать внешний IP.
2. Узнать страну.
3. Узнать ASN.
4. Проверить, слишком ли часто меняется страна между сессиями.
5. Отметить, что split tunneling ломает наивную проверку по одной активной сети.

### Проверка локального SOCKS

Нужно уметь:

1. Проверить стандартные loopback-порты.
2. Попробовать анонимный SOCKS-handshake.
3. Если в приложении включён локальный SOCKS и есть тестовые данные, проверить авторизованный доступ.
4. Явно показать:
   1. открыт ли порт,
   2. требует ли авторизацию,
   3. опасен ли режим.

### Проверка split tunneling

Нужно уметь показать разницу между:

1. доверенным приложением,
2. недоверенным приложением,
3. режимом обычного VPN,
4. режимом приватной сессии.

Итоговый отчёт должен отвечать на вопросы:

1. Может ли обычное приложение увидеть признаки VPN через Android API.
2. Есть ли сеть у недоверенного приложения.
3. Видит ли доверенное приложение VPN-egress IP.
4. Палится ли локальный SOCKS.
5. Корректно ли работает приватная сессия.

Что умеет:

1. Выбор DNS внутри VPN.
2. Пресеты: Cloudflare / Google / Quad9 / пользовательский.

## 11. Требования безопасности

## 11.1 Обязательные требования

1. Не использовать опасные локальные inbound по умолчанию.
2. Не использовать `allowBypass()`.
3. Явно задавать DNS внутри туннеля.
4. Если в импортированном профиле DNS отсутствует, приложение обязано подставить безопасный DNS по умолчанию само.
5. Не хранить секреты в логах.
6. Маскировать чувствительные данные в UI.
7. Не экспортировать секреты без явного подтверждения пользователя.

## 11.2 Правила самопроверки перед каждой сессией

1. Есть активный профиль.
2. Есть список доверенных приложений.
3. DNS настроен.
4. Источник DNS определён и показан пользователю.
5. Нет опасного локального inbound.
6. Пользователь предупреждён, если `Блокировать подключения без VPN` выключен.

## 11.3 Модель угроз

Недоверенными считаются:

1. банковские приложения
2. маркетплейсы
3. госприложения
4. приложения с агрессивной телеметрией
5. любые приложения вне списка доверенных

Доверенными считаются только приложения, которые пользователь явно добавил в allowlist.

## 12. План проверки и тестовый стенд

Нужно не просто сделать VPN-клиент, а сразу заложить **проверочный стенд**.

## 12.1 Что именно надо проверять

Нужно покрыть практические векторы детекта:

1. внешний IP
2. страна и ASN IP
3. репутационные эвристики
4. чтение сетевого состояния через Android API
5. локальный SOCKS на loopback
6. split tunneling
7. режим «Приватная сессия», когда недоверенные приложения должны остаться без сети

## 12.2 Состав стенда

### A. Встроенная лаборатория сканера

Для ручной проверки внутри основного приложения.

### B. Отдельное тестовое Android-приложение

Минимальное приложение-«сканер».

Что оно должно уметь:

1. вызывать `ConnectivityManager`, `NetworkCapabilities`, `LinkProperties`
2. делать запросы на тестовый backend и видеть внешний IP
3. пытаться подключаться к localhost SOCKS
4. показывать отчёт в UI

### C. Тестовый backend

Нужен простой сервер с endpoint'ами:

1. `/ip`
2. `/geo`
3. `/asn`
4. `/blacklist-check`
5. `/echo`

## 12.3 Обязательные тест-кейсы

### TC1. Обычный VPN, полный туннель

Ожидание:

1. Тестовое приложение видит VPN-признаки через Android API.
2. Внешний IP тестового приложения равен VPN-egress IP.

### TC2. Split tunneling: доверенное приложение через VPN, сканер вне VPN

Ожидание:

1. Доверенное приложение видит VPN-egress IP.
2. Сканер видит direct IP.
3. Сканер не должен автоматически узнать VPN-egress IP, если нет побочного канала.

### TC3. Приватная сессия + блокировка без VPN

Ожидание:

1. Доверенное приложение имеет сеть через VPN.
2. Сканер вне allowlist не имеет сети.
3. Сканер не может узнать внешний IP, потому что не может выйти наружу.

### TC4. Локальный SOCKS выключен

Ожидание:

1. На стандартных loopback-портах ничего опасного не слушает.

### TC5. Локальный SOCKS включён в безопасном режиме

Ожидание:

1. Порт открыт.
2. Анонимное подключение не проходит.
3. Авторизованное тестовое подключение проходит.
4. В отчёте это помечается как допустимый power-user режим.

### TC6. Локальный SOCKS включён в небезопасном режиме

Ожидание:

1. Порт открыт.
2. Анонимное подключение проходит.
3. Проверка безопасности помечает это как критическую проблему.

### TC7. DNS внутри туннеля

Ожидание:

1. DNS идёт через заданные DNS внутри VPN.
2. При расхождении показывается предупреждение или ошибка.

### TC8. Импорт raw Xray JSON без блока dns

Ожидание:

1. Профиль успешно импортируется.
2. Подключение не ломается только из-за отсутствия `dns`.
3. Приложение автоматически подставляет DNS по умолчанию.
4. В UI источник DNS отображается как `По умолчанию приложения`.
5. Пользователь может поменять DNS вручную без редактирования исходного JSON.

## 13. Acceptance Criteria для MVP

MVP считается успешным, если:

1. Приложение собирается и открывается в Android Studio без ручных танцев.
2. Поддерживается импорт `vless://...`.
3. Поддерживается импорт raw Xray JSON формата, аналогичного desktop-конфигам.
4. Работает подключение по **VLESS + REALITY**.
5. Работает режим «Обычный VPN».
6. Импорт raw Xray JSON без блока `dns` не ломает профиль и не требует ручной правки JSON.
7. Приложение явно показывает, какой DNS используется: из профиля, по умолчанию приложения или пользовательский.
6. Работает режим «Приватная сессия» через allowlist приложений.
7. Пользователь может включить `Всегда включённый VPN` и `Блокировать подключения без VPN` через понятный UX.
8. Встроенная лаборатория сканера показывает практический результат.
9. Без root и без Shizuku приложение работает как основной пользовательский сценарий.
10. Весь интерфейс русифицирован.

## 16. Ключевое правило для Codex

Не скатываться в попытку «магически скрыть VPN от всей системы».

Нужно построить **честный, практичный и рабочий Android-продукт**, который:

1. хорошо работает как Xray/VLESS/REALITY клиент,
2. даёт режим приватной сессии через allowlist,
3. режет сеть недоверенным приложениям через штатную Android-механику,
4. сам умеет проверять практические утечки.
